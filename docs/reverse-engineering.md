# Reverse-engineering notes

These notes track observed asset-format behaviour from locally supplied original game data. They are not a specification yet; they are working notes for clean-room decoding.

## Palette data

- Candidate files include `HPALxx.DAT`, `HPALETTE.DAT`, and `COL01.DAT` under `SYNDICAT/DATA` and `DATADISK/DATA`.
- The current decoder treats the first 768 bytes as a 256-colour VGA palette with 6-bit RGB channels.
- The HUD displays a 32-colour ramp sampled from the decoded palette.
- Current observed palette candidates begin with an `RNC` signature, meaning they are compressed/enveloped rather than direct palette payloads.
- RNC method-1 decompression is implemented with CRC-16/IBM verification for both packed payloads and unpacked output.
- `HPAL02.DAT` is preferred for the current MAP01 runtime render because FreeSynd's campaign data associates mission 1 with palette id 2. `HPALETTE.DAT` remains a generic fallback palette candidate; `COL01.DAT` decompresses to a smaller 256-byte tile-type table and is tracked separately from full VGA palettes.

## RNC containers

- Header fields are big-endian for packed/unpacked lengths and CRCs; payload data begins after the 18-byte header.
- Method 1 uses LSB-first Huffman/LZ blocks and header byte 17 as the block count. Header byte 16 is retained as leeway/in-place metadata but is not needed for out-of-place decoding.
- The decoder verifies packed CRC before decompression and unpacked CRC before returning bytes. Method 2 remains unsupported until a clean fixture and implementation plan are available.

## MAP data

- All observed `MAP*.DAT` files are RNC method-1 containers and now decompress with packed/unpacked CRC verification.
- `MAP01.DAT` is now parsed at runtime as original tile-stack data: three little-endian dimensions (`128 x 96 x 12` in the local first-level sample), followed by `width * depth` little-endian offsets measured from byte 12, followed by packed tile-index stacks. Each stack contributes up to `height` HBLK tile indices for a map column, in file order from lower to higher Z. This runtime parser feeds the first original-level render path; it does not write decoded map bytes or previews to source, docs, reports, or tests.
- The runtime original-map path now reads metadata-only selection data from local `GAME##.DAT` using FreeSynd's documented `mapinfos` offset (`113960`) for the selected campaign block. The current boot block derives mission id 1 and palette id 2 from the FreeSynd campaign table, then reads map id and scroll bounds from local mission data before loading `MAP##.DAT` and `HPAL##.DAT`. This selection layer does not decode objectives, people, vehicles, statics, weapons, scenarios, mission logic, or gameplay semantics.
- The original-map renderer now mirrors FreeSynd's low-Z map-boundary fallback for visual fill: out-of-bounds map coordinates draw tile id 6 only on Z levels 0 and 1, while higher out-of-bounds levels remain empty. This is a renderer edge-fill behavior only; it is not terrain classification, walkability, object placement, or mission semantics.
- Original-map camera startup and panning now use a dedicated viewport helper that projects between FreeSynd-style scroll-tile coordinates and the local renderer's world-space viewport origin. The viewport origin is clamped to the selected mission's `min_scroll_tile`/`max_scroll_tile` aggregate metadata after pan/zoom updates, reducing invalid edge exposure while preserving the existing runtime MAP/HBLK tile renderer.
- `GAME##.DAT` fixed sections are now summarized for aggregate-only render planning diagnostics using FreeSynd's section layout and object draw queue order: people, vehicles, weapons, statics, and sfx are queued after terrain on matching tiles. The diagnostics report record counts, candidate draw-record counts, type/subtype cardinalities, and tile coordinate ranges only. They do not render objects, decode sprites, output per-object positions or previews, prove entity semantics, or switch gameplay/pathfinding away from the demo grid.
- The runtime now also builds a typed first-mission scene model from local `GAME01.DAT` using the FreeSynd `LevelData` fixed-section offsets: people, vehicles, statics, weapons, sfx, scenarios, GAME objectives, and the 128x128 map object-link table. Candidate records keep tile/off/Z placement, draw-stage, type/subtype/orientation, animation/frame references, typed objective target buckets, scenario action/trigger chain buckets, guarded weapon/loadout hints, and debug-only interaction/action preconditions in memory for runtime use. HUD and generated report output remain aggregate-only: section counts, queue health, viewport-visible candidate totals, spawn/navigation bridge counts, candidate occupied/blocking buckets, interaction/objective/scenario buckets, weapon-loadout hint buckets, debug objective-progress labels, animation/frame support counts, HSPR sprite-bank support counts, guarded object-render proof counts, aggregate spatial/route/debug-navigation/action probe counts, and guardrail labels. No raw bytes, chunks, per-object dumps, decoded dimensions, anchors, previews, screenshots, objective logic, or gameplay semantics are written to source, docs, reports, or tests.
- Object rendering now has a guarded runtime proof path. The current gate decodes local six-byte `HSPR-0.TAB` entries with RNC-unwrapped `HSPR-0.DAT` payloads into an in-memory RGBA atlas using the selected mission palette, assembles local `HSTA-0.ANI`/`HFRA-0.ANI`/`HELE-0.ANI` frame-element chains with offsets and flip flags, and renders only object candidates whose frame and sprite references pass strict runtime bounds checks. Statics keep their existing proven path; peds use conservative FreeSynd-style standing/walking/dead directional animation candidates; vehicles and ground weapons have separate guarded proof paths and render only when their aggregate proof passes. The current renderer follows FreeSynd's object anchor model: the map renderer supplies the tile/base screen point, object sub-tile offsets are applied by the object draw step, and vehicle draws add the known extra vertical adjustment before applying offsets. Controlled original-agent sprites are special-cased to use the current route/base anchor rather than stale GAME-record sub-tile offsets, so route markers and ped feet stay aligned while local control moves the marker. Generated docs/reports keep this aggregate-only: candidate counts, frame-assembly counts, render-readiness labels, spatial/route-probe buckets, navigation-probe buckets, and guardrails, with no raw bytes, chunks, decoded dimensions, anchors, screenshots, or sprite previews. Sfx, objective logic, AI, final walkability, and gameplay/pathfinding remain candidate-only.
- `COL01.DAT` is decoded at runtime as a 256-byte tile-type table for semantics and region targeting, not as a draw visibility table. Runtime tile visibility comes from decoded HBLK alpha/visible-pixel data, matching the FreeSynd renderer's separation between tile pixels and tile type. The current spatial model uses those tile types to build a conservative 8-direction route graph, applies diagonal corner blockers, removes candidate static and vehicle footprint tiles, tracks ped occupancy and player/enemy spawn groups, counts candidate large-door surface patches from FreeSynd-style static subtype/orientation rules, distinguishes road/roof/train/safe-walk candidate nodes in aggregate diagnostics, and blocks normal route probes that require unproven height/slope transitions. The first-mission control view enables `G`-gated original-agent markers by default on entry, pre-snaps the four local original-agent control markers to distinct nearby proven route surfaces before the first rendered frame, hides only the controlled base-scene ped records while leaving non-squad ped candidates visible with FreeSynd-style type-label overlays, can traverse candidate slope/height edges, supports multi-select movement and Shift-right-click route queueing, replaces completed routes without replaying previous starts, and keeps local action/combat state separate from source GAME data and final gameplay mutation. Local combat currently derives a current objective/non-squad ped target candidate from typed GAME/MISS records, attaches guarded current-weapon/equipment/inventory-owner hints where GAME offsets support them, falls back to FreeSynd's starter pistol for the four local player markers, uses FreeSynd reference weapon range/reload data for local shot gates, shows temporary shot/status overlays, and tracks runtime-only HP/down/objective-complete state for debugging. It does not replace demo gameplay/pathfinding by default and is still not a final walkability, door/window, vehicle-footprint, burst/accuracy, AI, objective, or object-semantics decoder.
- The older aggregate diagnostic analyzer still slices the first `64 * 64 * 12 = 49152` bytes as conservative 12-byte comparison records. That slice overlaps the real MAP offset table and is retained only as a non-reconstructable cross-map signal surface until the generated report is migrated to the real tile-stack model.
- Remaining decoded bytes form a variable tail. Observed tails are aligned to 12-byte records, suggesting additional map/object records, but those records are not semantically decoded yet.
- The generated report lists primary-cell uniqueness, empty-cell counts, and tail record counts as aggregate diagnostics only; it does not include asset bytes.
- The HUD can render an abstract 64x64 cell-signature preview for `MAP01.DAT`. Colours represent frequency-ranked exact 12-byte cell signatures, not decoded terrain types.
- The analyzer now also reports per-word ranges and candidate low-byte lanes (`b0`, `b4`, `b8`) for the three 32-bit words, plus top byte-value counts. These are aggregate field diagnostics only.
- A gated inferred-layer preview derives low-risk visual channels from dominant byte-lane baselines: word-0 low byte as a surface candidate, word-1 low byte as a detail candidate, word-2 low byte as a reference candidate, and the narrowest varying high byte as a height-like candidate. These labels are deliberately provisional and should not be treated as final terrain/building semantics.
- The report now adds evidence-backed field-correlation diagnostics across decoded MAP primary sections: global byte-lane distributions, neighbour-continuity percentages, common value transitions, repeated 2x2/block-like patterns, and conservative height-gradient checks. These diagnostics are intended to rank candidate fields; they are not claims of exact terrain/building semantics.
- A provisional `MapPrimarySubstrateCandidate` now copies the selected diagnostic byte lanes into non-gameplay substrate channels: `surface_index_candidate`, `detail_index_candidate`, `reference_candidate`, and `height_candidate`. Each channel records its selected lane, dominant baseline, unique count, continuity, repeated-block evidence, gentle-gradient evidence, and a low/medium/high heuristic confidence label. These confidence labels summarize evidence strength only and do not prove terrain, building, or object semantics.
- BLK-like graphics containers such as `MMAPBLK.DAT`, `HBLK01.DAT`, `MMAP.DAT`, and `MMAPOUT.DAT` are now inspected with non-reconstructable aggregate diagnostics: RNC status, decoded length, byte entropy/zero/unique summaries, plausible fixed-size indexed-pixel record counts, aggregate fixed-record layout probes, duplicate-record counts via checksums, per-record zero/unique/entropy min/median/max summaries, conservative leading/trailing table or remainder hints, and range-only correlations against MAP substrate candidate byte lanes. These rows can show that a MAP candidate range could address a block/tile candidate record count or that a container has very low aggregate entropy, but they do not prove terrain, building, object, minimap, mask, or render-layout semantics.
- The generated report also includes cross-container aggregate relation probes for BLK-like and TAB/DAT candidates. These compare decoded/plain length ratios, duplicate-name decoded length/hash status between `SYNDICAT/DATA` and `DATADISK/DATA`, layout-alignment support rankings across candidate dimensions, and TAB/DAT chunk-count compatibility against BLK-like fixed-record counts. These probes use only counts, lengths, entropy summaries, and non-reconstructable hash/status comparisons; they are intended to prioritize future investigation and do not establish render formats or semantic links.
- Press `M` in the prototype to cycle the main world view between the runtime original mission-map tile renderer, first-mission control view, playable hand-authored demo city, and a first-class runtime-local decoded `MAP*.DAT` diagnostic scene catalog. With local `GAME##.DAT`, `MAP##.DAT`, `HBLK01.DAT`, `COL01.DAT`, and a local mission VGA palette available, startup now opens directly on the selected runtime original map render, framed on the largest connected non-fill region and clamped to mission scroll bounds so the first-level compound is visible without manual panning. Press `N`/`P` to select the next or previous decoded MAP scene for diagnostics. Scene layers include signature classes, the provisional inferred layer, individual candidate-field explorer views for `surface_index_candidate`, `detail_index_candidate`, `reference_candidate`, and `height_candidate`, an aggregate block-addressability overlay, a runtime original-graphics candidate MAP view, and a runtime original-graphics atlas view. Candidate-field views show the selected lane, confidence label, unique-value count, continuity percentage, and repeated-2x2 percentage in the diagnostic panel. The block overlay ranks MAP candidate fields against capped block/tile record-count candidates and colors cells by whether the selected field value could address the selected aggregate record count. The original-graphics atlas decodes `HBLK01.DAT` at runtime as 256 local 64x48 map-tile records assembled from six 32x16 packed subtiles per tile, remaps the 4-bit colour planes through the mission palette, applies the original-style tile-buffer orientation, uploads the RGBA atlas to Macroquad, and preserves the tile aspect ratio on screen. The original mission-map renderer draws runtime HBLK tiles from decoded MAP stack placement using file-order Z levels, HBLK alpha visibility, and a viewport-bounded traversal modeled on FreeSynd's `MapRenderer::render` loop; the first-mission control view then draws proofed original object sprites after terrain on matching queued tiles while keeping candidate-only categories and gameplay semantics visibly separate. In first-mission control mode, hovering identifies a candidate original-map cursor tile with local sub-tile offsets, right-click moves selected local original-agent markers along proven candidate routes, Shift-right-click appends to their movement queues, `F` focuses the camera on the selected marker, left-click selects nearby markers or has the selected original-agent set fire at the clicked current-objective/non-squad ped candidate, and `E` queues route-gated interaction/action intents for aggregate door/pickup/vehicle/objective/scenario buckets. Non-squad ped candidates get role overlays such as target, civilian, guard, police, criminal, or candidate agent, and local combat clicks briefly draw shot/status lines for hit, blocked, range, cooldown, or down outcomes. Reaching an interaction target advances local action state to resolved labels for the safest candidate classes. Local combat may mark a runtime-only ped target down and a runtime-only objective complete, but it does not mutate source GAME data, final mission results, AI, doors, inventory, vehicles, or switch pathfinding off the demo grid. No asset-derived pixels or previews are written to source, docs, reports, or tests. Local screenshots for visual comparison must be written only to ignored `visual_diagnostics/` or another untracked temporary path. These views do not prove final walkability, door/window behavior, object fields, entity placement semantics, objectives, combat, or switch gameplay/pathfinding off the demo tactical grid. Tactical agents, hostiles, path overlays, and the minimap are intentionally hidden in decoded MAP and original-graphics candidate modes so the view does not imply final gameplay semantics.

## TAB/DAT banks

Observed `.TAB` files are not all the same shape:

- Some banks look compatible with 32-bit little-endian offsets into paired `.DAT` files.
- Some files have odd byte lengths or patterns that suggest packed records, 16-bit fields, flags, dimensions, or mixed metadata rather than a plain offset table.
- The engine now scores 16/24/32-bit little-endian interpretations by valid offset count, unique offset count, and monotonic adjacent pairs.
- Safely parsed 32-bit TAB/DAT archives now report aggregate-only chunk diagnostics: chunk count, min/median/max chunk size, common chunk-size buckets, chunk-size entropy, duplicate-offset/zero-length candidate counts, first/last offset sanity ranges, exact matches to fixed tile-byte candidates such as 64/256/512/1024/2048/4096 bytes, chunk-length progression candidates (equal-size runs, common adjacent length deltas, and repeated size-pattern counts), aggregate sprite chunk classifier counts, classifier distribution by chunk-size buckets, small/medium/large chunk count bands, zero/high-byte ratio min/median/max summaries by classifier kind, candidate leading-byte/header-shape frequency counts, and conservative candidate metadata-shape support counts/ranges for bounded leading dimension/offset interpretations. These summaries do not include chunk bytes or render previews.
- Cross-container probes compare TAB/DAT chunk-size distributions and chunk counts with BLK-like fixed-record candidates. The report also groups safely parsed TAB/DAT archives by conservative file-name families such as `HSPR`, `MSPR`, `MFNT`, `FONT`, and `SOUND`, then compares only aggregate classifier totals and candidate header-shape counts. These are compatibility rankings for future investigation only; they do not prove that a TAB/DAT bank is a sprite, tile, font, sound, UI, or map-support format.
- The generated report now includes dedicated TAB/sprite family aggregate-ranking, next-investigation hint, capped per-archive evidence, investigation-dashboard, runtime-probe planning, cross-family runtime-probe queue, runtime probe selector catalog, local dry-run ordering, runtime probe manifest summary/phases, local runtime probe execution diagnostics/phases, and comparison sections for safely parsed TAB/DAT archives. The ranking section uses only parsed archive counts, total chunk counts, command-stream/raw/unknown classifier totals, bounded candidate metadata-shape support ratios, equal-size run and repeated size-pattern support, chunk-size entropy ranges, and overlapping common chunk-size buckets. The hint section turns those same aggregate values into conservative per-family inspection priorities: strongest candidate metadata-shape to inspect next, strongest classifier/size-band pattern to prioritize, whether chunk-length progression suggests repeated or varied records, and whether entropy/common-bucket behaviour looks closer to command-stream-heavy, compact metadata/control-record, or mixed banks. The per-archive evidence section expands only the top selected sprite-like family hints into capped archive rows using archive paths, chunk counts and size bands, candidate metadata-shape support counts/ranges, classifier totals by size band, chunk-length progression clues, entropy, and common-size bucket summaries. The investigation dashboard combines capped within-family consistency, selection rationale, and conservative runtime-only next probes for the top selected sprite-like families: archive inclusion/capping status, chunk-count/range consistency, shared metadata-shape candidates and support-ratio spread, classifier-by-size-band consistency, common bucket overlap/distinction, entropy spread, progression agreement/disagreement, and next clean-room probes such as repeated fixed-length command/control-record candidates, candidate leading offset-pair metadata shapes, command-stream-heavy separation before rendering attempts, and sibling common-bucket comparison. The runtime-probe planning section converts those top dashboard families into capped aggregate-only local probe plans by grouping repeated chunk-length buckets, candidate leading metadata-shape support, command-stream-heavy versus mixed/raw/unknown classifier totals, and sibling archive common-size bucket comparisons. The cross-family runtime-probe queue and selector catalog are built from a shared aggregate workbench model: selector IDs are stable slugs derived only from family candidate, probe category, support tier, and capped rank, while selector rows carry runtime preconditions and stop conditions for local user-supplied assets. The local dry-run ordering, runtime manifest summary, and aggregate execution diagnostics group selectors into metadata-shape grouping, command-stream-heavy separation, repeated fixed-length bucket grouping, sibling/common bucket comparison, and fallback mixed/unknown audit phases, then report execution readiness plus aggregate group/support counts only. The comparison section selects the top sprite-like filename-family candidates from that ranking, caps output to the top three selected families to avoid noisy all-pairs listings, and compares those pairs using aggregate ratio differences, progression-support differences, entropy ranges, overlapping/distinct common chunk-size buckets, strongest shared compatibility clues, and strongest distinguishing clues. These rows are prioritization aids for future clean-room decoding and do not render sprites, expose chunk/header bytes, decoded dimensions, anchors, commands, graphics, font, sound, UI, mission, or gameplay semantics.
- `engine::runtime_probe::TabRuntimeProbeManifest` is now a runtime-facing version of that capped aggregate workbench, and `TabRuntimeProbeExecution` runs a local-only aggregate dry-run over those selectors. `AssetIndex` builds the manifest from local user-supplied original assets, `cargo run --bin probe_manifest -- ../original_assets` prints selector planning, and `cargo run --bin probe_manifest -- --execute ../original_assets` prints execution readiness, aggregate group counts, aggregate support counts, phase totals, and stop conditions without generating the full report. This is a clean-room decoder queue surface only; it does not decode or render sprites and does not write reconstructable asset-derived output.

Current conservative approach:

1. Use `TabArchive` only when a bank can be parsed safely into bounded chunks.
2. Use `TabVariantAnalysis` to report which offset width looks most plausible.
3. Use aggregate summaries, runtime probe manifests, aggregate dry-run executions, and synthetic tests to rank candidate relationships without exposing reconstructable data.
4. Keep generalized sprite rendering disabled until each bank's layout is proven; the current exception is the guarded runtime-local `HSPR-0`/ANI first-mission object path used by the first-mission control view.

## Sprite chunks

`sprite_decode.rs` currently classifies safe chunks as:

- empty,
- likely raw indexed pixels,
- likely RLE/command stream,
- unknown.

This remains diagnostic for unproven TAB/DAT families. `HSPR-0` game sprites now have a separate guarded runtime decoder for first-mission statics/peds/vehicles/ground weapons and original-agent markers where proof gates pass; future work should extend that proof toward sfx, stronger animation/state semantics, object occupancy, combat, and mission interaction.

Current aggregate sprite-bank diagnostics intentionally stop at non-reconstructable metadata: chunk counts, size buckets, classifier distributions, zero/high-byte ratio summaries, candidate header-shape frequencies, and candidate metadata-shape support counts/ranges. Header-shape and metadata-shape labels are provisional aggregate patterns only; they should not be interpreted as decoded sprite dimensions, anchors, commands, or audio metadata without stronger cross-file evidence.
